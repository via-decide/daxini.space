Database-less Storage Decision Report

This report documents the architectural findings of running a multi-user messaging system using the LLM's conversation history array as the sole storage mechanism.

CAUTION: Using the LLM context as a message relay is not recommended for production applications. It is slow, expensive, and subject to hard data loss when context limits are reached.

Comparison Matrix

Criteria LLM Thread Storage Lightweight DB (SQLite/Redis)
Reliability Extremely Low. Ollama crash or context pruning wipes history. High. Persistent storage with crash recovery.
Read latency ~1.5s - 3.5s (LLM parses history to extract transcript). <5ms. Direct indexed queries.
Message fidelity Unreliable. LLM might rewrite or summarize text. 100% exact message matches.
Cost curve Exponential scaling. Every single fetch re-processes all history. Flat scaling. Negligible RAM/Disk overhead.
Concurrent edits High risk of race conditions. No locking/ordering checks. Atomicity guarantees (ACID or memory lock threads).

Context Window Degradation

  • Exhaustion Curve: At the default context window limit of 2,048 tokens, the thread is exhausted after approximately 25-30 conversational turns.
  • Summarization Drift: When context size exceeds limits, older logs are dropped or summarized, changing historical context (e.g. "[A] Hello" becomes "[A] requested assistance").
  • Token Costs: Bandwidth grows by O(N²) where N is message count, since N messages are transferred and computed repeatedly.

Recommendation

Deploy the LLM solely for cognitive parsing (like intent analysis, translations, or safety filters) and route the actual message storage through a 20-line SQLite backend or Redis Key-Value store.
